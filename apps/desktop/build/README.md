# build/ - electron-builder resources

This directory contains the electron-builder resources for packaging SkillKeeper.

## Icons

`icon.icon` is the source, authored in Apple's Icon Composer (Liquid Glass:
layered sparkle glyph over a gradient). Export it from Icon Composer as two
full-bleed 1024x1024 PNGs -- the only committed rasters:

- `icon-default.png` -- light appearance (the app's default icon)
- `icon-dark.png` -- dark appearance (used at runtime under a dark OS theme)

Icon Composer exports a full-bleed square (the rounded body touches the canvas
edges, as iOS expects). Desktop icon slots -- the macOS dock especially --
expect the artwork inset within the canvas (Apple's macOS grid: an 824x824 body
in a 1024 canvas), or the icon looks oversized next to its neighbours. So all
platform outputs are generated from the sources by
[`generate-icons.mjs`](./generate-icons.mjs), a cross-platform, pure-JS script
(no `iconutil` / `sips` / ImageMagick needed -- uses `png2icons` and `jimp`),
which first insets each source into a padded master:

- `icon.app.png` / `icon-dark.app.png` -- padded light/dark masters. The light
  one is the Linux icon (`build.linux.icon`); both are imported by
  `src/main/index.ts` and swapped at runtime to follow the app theme preference
  (`config.general.theme`: system/light/dark -- main mirrors it onto
  `nativeTheme.themeSource`). Packaged icons do not apply in `dev`, and the OS
  does not theme-swap an installed app's icon on its own, so the swap is done in
  the main process.
- `icon.icns` -- macOS dmg (`build.mac.icon`), from the light master
- `icon.ico` -- Windows nsis / appx (`build.win.icon`), from the light master
- `appx/*.png` -- Windows Store Visual Assets (StoreLogo, Square44/71/150/310,
  Wide310x150), from the light master

Icon Composer exports 16-bit PNGs (~1.5 MB). The script checks each source and,
if it is 16-bit, rewrites it to 8-bit in place (visually lossless, ~4x smaller)
*before* deriving anything -- so the committed sources stay small automatically.
This is the only case where running the script modifies a committed file, and
only right after a fresh re-export.

These generated files are **git-ignored** (see [`.gitignore`](./.gitignore))
and (re)created at build time: the `dev`, `build`, `package` and `package:dir`
scripts all run `generate-icons` first, so the binaries are always present and
fresh. Generation is **skipped when `icon.png` is unchanged** -- a sha256 of the
source is cached in `.icon-cache.json` (also git-ignored) and generation only
runs when the hash differs or an output is missing (e.g. a fresh checkout). So
the pre-`dev` hook is effectively a no-op after the first run.

To regenerate manually:

    pnpm --filter @skillkeeper/desktop run generate-icons          # skip if unchanged
    pnpm --filter @skillkeeper/desktop run generate-icons -- --force

Re-export `icon.png` from Icon Composer whenever the design changes; the next
`dev`/`build`/`package` picks it up automatically (hash changes).

## macOS signing

`entitlements.mac.plist` carries the hardened-runtime entitlements needed by
Electron (JIT) and the `node-pty` native module (library-validation). Wired via
`build.mac.entitlements` / `entitlementsInherit` with `hardenedRuntime: true`.

File access: the dmg build is not sandboxed, so no `com.apple.security.files.*`
entitlements are needed. Access to TCC-protected folders (Documents, Desktop,
Downloads, removable volumes) is covered by the `NS*UsageDescription` strings in
`build.mac.extendInfo`. A sandboxed entitlements file would only be needed for a
Mac App Store (`mas`) target.

## Linux desktop entry

`build.linux.desktop.entry` in `package.json` defines the generated `.desktop`
file (Name, Comment, Categories, Keywords, StartupWMClass) -- no separate file
to maintain.

## Windows Store

The AppxManifest is generated by electron-builder from the `build.appx` block
in `package.json`; only the Visual Assets under `appx/` are supplied (and those
are generated by the script above). `appx/` will also hold any hand-authored
Store metadata added later as part of the release pipeline.

## electron-builder config

The `build` field in `package.json` contains the electron-builder configuration.
